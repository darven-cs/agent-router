# Go 轻量本地 API 中转方案（Claude Code 专属）

> 完全匹配你的需求：**本地运行、轻量无依赖、适配 3 家厂商 API、统一输出 Claude Code 标准 API、负载均衡、自动故障切换、用量监控、TUI 可视化界面**

基于 Go 原生 + 极简第三方库开发，**单文件可运行**，数据本地存储，无外部服务依赖。

## 一、核心定位

### 1.1 统一出口

对外暴露 **标准 Claude API**（`/v1/messages`），仅暴露 `claude-opus-4.6` 模型（你指定的核心模型）

### 1.2 适配上游

无缝对接 3 家 Anthropic 兼容 API：

| 厂商 | 端点 | 鉴权方式 |
|------|------|----------|
| 智谱 | `https://open.bigmodel.cn/api/anthropic` | `x-api-key` / Bearer Token |
| Aicodee | `https://v2.aicodee.com` | `x-api-key` / Bearer Token |
| Minimax | `https://api.minimax.com/anthropic` | `x-api-key` / Bearer Token |

### 1.3 核心能力

- **负载均衡**：取模算法，简单高效
- **故障切换**：请求失败自动切换下一个上游
- **配置热更新**：SIGHUP 信号 / TUI 按钮 / 动态添加删除上游
- **用量监控**：本地 SQLite 持久化 + TUI 实时展示

### 1.4 轻量特性

单二进制文件 | 本地 SQLite 存储 | 内存状态管理 | 启动即用

| 模块 | 技术方案 | 原因 |
|------|----------|------|

## 四、核心功能

### 4.1 负载均衡（取模算法）

#### 算法原理

```
1. 将所有启用的上游渠道按顺序编号（0, 1, 2, ...）
2. 根据请求特征计算 hash 值（使用请求 ID 或客户端 IP）
3. target_index = hash % enabled_channels_count
4. 直接路由到对应渠道
```

#### 示例

```
假设启用渠道：[zhipu, aicodee, minimax]（共 3 个）

请求 A：hash = 100，100 % 3 = 1 → 路由到 aicodee
请求 B：hash = 101，101 % 3 = 2 → 路由到 minimax
请求 C：hash = 102，102 % 3 = 0 → 路由到 zhipu
```

#### 优势

| 特性 | 说明 |
|------|------|
| **O(1) 效率** | 无序遍历，直接计算 |
| **哈希均匀** | 请求分布均衡 |
| **简单可靠** | 无状态，易于理解和维护 |

### 4.2 故障切换

#### 触发条件

- 上游响应 5xx 错误
- 上游连接超时
- 上游返回格式错误

#### 切换逻辑

```
1. 当前渠道请求失败
2. 按取模算法选择下一个渠道
3. 应用指数退避重试（1s → 2s → 4s）
4. 达到最大重试次数（默认 3 次）仍失败
5. 切换到下一个渠道继续尝试
6. 所有渠道都失败，返回错误
```

#### 重试参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 最大重试次数 | 3 | 单次请求最大重试次数 |
| 初始退避时间 | 1 秒 | 第一次重试等待时间 |
| 最大退避时间 | 4 秒 | 最长等待时间 |
| 退避策略 | 指数退避 | 1s → 2s → 4s |

### 4.3 用量监控

#### 存储结构

- **存储介质**：本地 `usage.db` SQLite 文件
- **写入方式**：异步 goroutine 写入，不影响主请求性能

#### 统计维度

| 维度 | 说明 |
|------|------|
| 总请求数 | 累计成功 + 失败请求数 |
| 成功/失败数 | 按状态分组的请求计数 |
| 输入令牌 | anthropic-input-tokens 累计 |
| 输出令牌 | anthropic-output-tokens 累计 |
| 总令牌 | 输入 + 输出累计 |
| 各渠道调用量 | 按上游分组的请求计数 |
| 各渠道成功率 | 各上游成功/失败比例 |

## 六、API 接口

### 6.1 标准 Claude API（对外暴露）

#### POST /v1/messages

**请求头（二选一）**

```bash
# Bearer Token 形式
Authorization: Bearer <your-local-proxy-key>

# 或 x-api-key 形式
x-api-key: <your-local-proxy-key>
Content-Type: application/json
```

**请求体**（完全兼容 Claude 官方 SDK）

```json
{
  "model": "claude-opus-4-6",
  "max_tokens": 1024,
  "messages": [
    {"role": "user", "content": "写一个快排算法"}
  ]
}
```

**响应体**（标准 Claude 响应）

```json
{
  "id": "msg_xxx",
  "type": "message",
  "role": "assistant",
  "content": [
    {"type": "text", "text": "..."}
  ],
  "model": "claude-opus-4-6",
  "stop_reason": "end_turn",
  "usage": {
    "input_tokens": 1024,
    "output_tokens": 2048
  }
}
```

### 6.2 管理 API（内部使用）

#### POST /admin/reload

重新加载配置文件（触发配置热更新）

**请求头**

```
Authorization: Bearer <admin-key>
```

**响应**

```json
{
  "status": "ok",
  "message": "Configuration reloaded",
  "channels": 3
}
```

#### GET /admin/status

获取服务状态

**响应**

```json
{
  "status": "running",
  "uptime": 9015,
  "channels": {
    "total": 3,
    "enabled": 2,
    "disabled": 1
  },
  "usage": {
    "total_tokens": 1234567,
    "total_requests": 45231,
    "success_rate": 0.999
  }
}
```

### 6.3 错误响应

```json
{
  "error": {
    "type": "upstream_unavailable",
    "code": 1001,
    "message": "All upstream channels are unavailable"
  }
}
```

| 错误码 | 类型 | 说明 |
|--------|------|------|
| 1001 | upstream_unavailable | 所有上游不可用 |
| 1002 | upstream_timeout | 上游请求超时 |
| 1003 | upstream_bad_response | 上游返回错误格式 |
| 2001 | invalid_request | 请求格式错误 |
| 2002 | unauthorized | 鉴权失败 |
| 3001 | internal_error | 内部错误 |

## 七、配置说明

### 7.1 配置文件（config.yaml）

```yaml
# 本地中转服务配置
server:
  port: 8080
  api_key: "your-local-proxy-key"      # 本地 API 鉴权密钥
  admin_key: "your-admin-key"          # 管理接口鉴权密钥（可选）

# 上游渠道配置
upstream:
  - name: "zhipu"                       # 渠道名称（唯一标识）
    url: "https://open.bigmodel.cn/api/anthropic"
    api_key: "${ZHIPU_API_KEY}"         # 支持环境变量引用
    auth_type: "bearer"                 # 鉴权类型：bearer / x-api-key
    weight: 5                           # 权重（影响取模路由）
    enabled: true                       # 启用/禁用
    timeout: 30                         # 超时时间（秒）

  - name: "aicodee"
    url: "https://v2.aicodee.com"
    api_key: "${AICODEE_API_KEY}"
    auth_type: "x-api-key"
    weight: 3
    enabled: true
    timeout: 30

  - name: "minimax"
    url: "https://api.minimax.com/anthropic"
    api_key: "${MINIMAX_API_KEY}"
    auth_type: "bearer"
    weight: 4
    enabled: false                      # 默认禁用
    timeout: 30
```

### 7.2 配置项说明

| 配置项 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| server.port | int | 是 | 监听端口 |
| server.api_key | string | 是 | 本地 API 鉴权密钥 |
| server.admin_key | string | 否 | 管理接口密钥 |
| upstream[].name | string | 是 | 渠道唯一名称 |
| upstream[].url | string | 是 | 上游 API 地址 |
| upstream[].api_key | string | 是 | 上游 API 密钥 |
| upstream[].auth_type | string | 否 | bearer / x-api-key，默认 bearer |
| upstream[].weight | int | 否 | 权重，默认 1 |
| upstream[].enabled | bool | 否 | 是否启用，默认 true |
| upstream[].timeout | int | 否 | 超时秒数，默认 30 |

### 7.3 环境变量

支持在配置文件中使用 `${ENV_VAR}` 引用环境变量：

```bash
export ZHIPU_API_KEY="your-zhipu-key"
export AICODEE_API_KEY="your-aicodee-key"
export MINIMAX_API_KEY="your-minimax-key"
```
